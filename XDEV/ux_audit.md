# VV CRM — UX-Аудит. Робота з сутностями та User Flows

> **Фокус:** Як система поводиться як CRM для щоденної роботи. Не технічні баги — а зручність, логіка, повнота.
> **Методологія:** Пройдено кожен user flow від початку до кінця: customer journey, repair lifecycle, sale flow, inventory management.

---

## 🔴 КРИТИЧНО ЗЛАМАНІ FLOWS

### F1. Ремонти — немає сторінки деталей / «Картки ремонту»

**Проблема:**
Клік на рядок ремонту відкриває `EditRepairForm` — форму редагування. Тобто **перегляду** ремонту не існує взагалі. Майстер не може просто **подивитися** на ремонт — він одразу в режимі редагування.

**Наслідки:**
- Немає місця де бачити повну хронологію ремонту
- Немає customer communication log (поле є в БД, але ніде не відображається)
- Немає фото стану пристрою при прийманні (поле `device_condition_photos` — є в БД, але не відображається)
- Технік і власник бачать одну й ту ж форму редагування, без розділення ролей

**Потрібно:** Окрема сторінка або expanded drawer «Картка ремонту» з ReadOnly-переглядом, і вже звідти — кнопка «Редагувати».

---

### F2. Продажі — немає списку продажів

**Проблема:**
У системі немає розділу «Продажі». Продажі видно тільки в Dashboard (останні 5), Фінансах (50 транзакцій), Клієнт (detail drawer) та Звітах.

Але **окремої сторінки `/admin/sales`** із таблицею продажів — немає. Не можна:
- Переглянути всі продажі за день/тиждень
- Знайти конкретний продаж за клієнтом
- Скасувати/повернути продаж
- Роздрукувати чек
- Бачити що саме купив клієнт (item breakdown)

**Це критично** — це основна операція в магазині.

---

### F3. Ремонти — «Видано клієнту» і оплата роз'єднані

**Проблема:**
Статус `handed_over` і `payment_status: paid` — це два **окремі** поля, які ніяк не зв'язані. Можна видати пристрій без оплати. Система не попереджає. Немає flow «Отримати оплату + видати».

**Наслідок:** Ремонт «Видано» але «Не оплачено» — і ніхто цього не бачить в момент видачі.

---

### F4. Клієнти — неможливо зателефонувати з CRM

Телефон клієнта відображається як текст (`font-mono`), але **не є `<a href="tel:...">`**. На мобільному пристрої — потрібно скопіювати, перейти в телефон, набрати. Для майстерні де все телефоном — критично.

## 🟠 СЕРЙОЗНІ UX-ПРОБЛЕМИ

### U1. Ремонти — відсутня Kanban-дошка / Pipeline view

Є таблиця ремонтів з фільтром по статусу, але немає **Kanban-дошки** для ремонтів (на відміну від девайсів де вона є і добре зроблена). Майстер не бачить pipeline. Девайси мають чудовий Kanban (Transit → Stock → Service), а ремонти — звичайну таблицю де статус змінюється через `<select>`.

---

### U2. Продажі — не можна продати кілька товарів за раз

`SaleForm` дозволяє продати **рівно один** товар: або один девайс, або один аксесуар, або одну послугу. Кошика немає. Клієнт купив iPhone + чохол + захисне скло = треба 3 окремі продажі. Це реально болить.

---

### U3. Аксесуари — немає кнопки «Продати» з таблиці

У таблиці аксесуарів є тільки «Редагувати» та «Видалити». Немає швидкої кнопки «Продати», як є у девайсів. Для продажу аксесуара треба йти на Дашборд або в SaleForm окремо.

---

### U4. Запчастини — немає зв'язку з ремонтами

Склад запчастин (`/admin/parts`) існує окремо від ремонтів. Коли в EditRepairForm вказуєш «собівартість деталей» — це просто число, не прив'язане до реального списання зі складу:
- Запчастина списана з каси (cost у ремонті)
- Але stock у `parts` залишається незмінним
- Немає автоматичного списання запчастин при закритті ремонту

---

### U5. Клієнти — «Середній чек» рахується неправильно

```
Середній чек = totalSpent / customers.length
```
Це середній spend per customer, а не середній чек. Середній чек = сума всіх продажів / кількість продажів.

---

### U6. Ремонти — «Дата готовності» ніяк не використовується

Поле `estimated_completion` заповнюється при створенні ремонту, але:
- Не відображається в таблиці ремонтів
- Не видно у EditRepairForm
- Немає підсвітки «Прострочено» якщо дата минула
- Немає нагадувань / alerts

Поле є в БД, є в формі, але **мертве** в UI.

---

### U7. Purchases — немає розкриття позицій закупівлі

В таблиці закупівель колонка «Позицій» показує число (1, 3, 5...) але по кліку нічого не відкривається. Не можна побачити що саме закупили. Немає Detail drawer для закупівлі.

---

### U8. Фінанси — транзакції без пагінації

`data-finance.ts` повертає останні **50 транзакцій** (`.limit(50)`). При активній роботі магазина це кілька днів. Немає пагінації, немає фільтра по даті, немає пошуку.

---

### U9. Партнери — `window.location.reload()` після дії

```typescript
const handleSuccess = () => {
  setIsModalOpen(false);
  window.location.reload(); // замість router.refresh()
}
```
Повне перезавантаження сторінки замість revalidation. Втрачається scroll position, відчуття «додатку» руйнується.

---

### U10. Ремонти — немає кнопки «Зателефонувати клієнту» в картці

EditRepairForm показує ім'я клієнта, але немає ні посилання на клієнта, ні кнопки дзвінка, ні навігації до профілю клієнта.

---

## 🟡 СЕРЕДНІ UX-ПРОБЛЕМИ

### M1. Аксесуари — немає попередження про низький залишок

Показується просто кількість без візуальної ознаки «критично мало» якщо `stock` близько до `min_stock`. Іконка попередження є тільки в `parts`. Для аксесуарів — тільки зелений/червоний (0 або не 0).

### M2. Девайси — немає фото в Kanban картці

`DeviceForm` дозволяє завантажити фото (`photo_urls`). Але в Kanban картці — фото не відображається.

### M3. Ремонти — пошук тільки по клієнту та пристрою

Не можна шукати по IMEI, по номеру телефону клієнта, по TTN. Майстер запам'ятовує IMEI.

### M4. EditRepairForm — «Авторозрахована ціна» показує неправильну змінну

```tsx
{isExternal && (
  <span>{price} грн</span>  // показує OLD price state, не computedPrice
)}
```
Компонент рахує `computedPrice` через `useMemo`, але preview відображає `price` (стара змінна).

### M5. Dashboard — список активних ремонтів задублений

На дашборді є два блоки з ремонтами:
1. «Статуси ремонтів» (grid 3×1)
2. «Активні ремонти» (grid 1×1)

Один і той же `repairs` масив відображається двічі. Займає 50% екрана дашборда.

### M6. Клієнти — немає фільтра по VIP статусу

Є колонка VIP, є значення (regular/silver/gold/platinum), але фільтра по VIP немає.

### M7. Покупки — немає зв'язку з постачальником в деталях

Показується ім'я постачальника, але немає посилання на картку постачальника.

### M8. Ремонти — ReadOnly статус після `handed_over` незворотній

Після `handed_over` або `cancelled` — статус нередагований. Якщо клієнт повернувся з претензією — неможливо повернути ремонт.

### M9. SaleForm — промокод не скидається при зміні товару

При зміні категорії товару `handleCategoryChange` скидає `discount = 0`, але `promoCode` і `partnerId` залишаються. Наступний submit зарахує відкат партнеру без знижки.

### M10. Фінанси — поле «Прибуток» рахується некоректно

```typescript
profit = totalSales - totalPurchases - totalExpenses;
```
Продажі техніки включають cost_price. Але `totalPurchases` також включає закупівлі цієї техніки. Подвійне відрахування собівартості.

### M11. Аксесуари — «Статус» завжди зелений при будь-якому залишку

Якщо в наявності 1 шт а `min_stock = 5` — зелений badge «1 шт». Немає попередження.

### M12. Ремонт — фото «обов'язкове» але перевірки немає

```tsx
<input type="file" required />
```
HTML `required` на file input не спрацьовує в більшості браузерів. Server action не валідує. Можна створити ремонт без фото.

## Пріоритетний план виправлень

### Sprint 1 — Критичні flows (3-5 днів)
1. **`/admin/sales` сторінка** — повний список продажів з пошуком та фільтром
2. **Картка ремонту** — окремий Detail View (не форма редагування при кліку)
3. **`tel:` посилання** на всіх телефонах клієнтів

### Sprint 2 — Важливі покращення (3-4 дні)
4. **Multi-item кошик** у SaleForm (найбільший UX-прорив)
5. **Kanban для ремонтів** (аналог до девайсів)
6. **Estimated completion** — підсвітка прострочених, показ у таблиці
7. **Списання запчастин** зі складу при закритті ремонту

### Sprint 3 — Зручності (2-3 дні)
8. Кнопка «Продати» в таблиці аксесуарів
9. Попередження low stock для аксесуарів
10. Пагінація / фільтр по даті в транзакціях
11. Фільтр по VIP у клієнтах
12. `router.refresh()` замість `window.location.reload()` у Partners
13. Detail drawer для закупівлі (список позицій)
14. Виправлення M4 (computedPrice vs price у EditRepairForm)
15. Виправлення M9 (скидання промокоду при зміні товару)
