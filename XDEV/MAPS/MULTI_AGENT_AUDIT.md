> **Дата:** 2026-06-14  
> **Методологія:** `multi-agent-brainstorming`  
> **Об'єкт аудиту:** Блок 5 (Продажі & Bento POS) та Блок 6 (Складський облік)

## 👥 1. Мультиагентні Точки Зору (Reviewer Perspectives)

### 1️⃣ Skeptic / Challenger (Аналіз ризиків та відмов)
* **Проблема транзакційності (Розсинхрон складу):**  
  У `createMultiSaleAction` списання товарів відбувається в циклі `for` через окремі HTTP-запити до Supabase. Якщо під час обробки 3-го товару виникає `Conflict` (наприклад, закінчився stock), Server Action викидає помилку. Проте перші два товари вже були успішно списані з бази даних! Оскільки транзакція не відкочується автоматично, склад залишків розсинхронізується, що призведе до розходжень у базі.
* **Брак блокування на Devices:**  
  Статус техніки (`devices`) оновлюється на `"sold"` звичайним апдейтом без оптимістичного блокування:
  ```typescript
  await supabase.from("devices").update({ status: "sold" }).eq("id", item.item_id);
  ```
  Якщо два продавці одночасно додадуть один і той самий девайс (наприклад, з одним IMEI) у кошик та оформлять продажі паралельно, обидва отримають статус "успішно", що призведе до дублювання продажу унікальної фізичної техніки.
* **Спрощений облік гарантії:**  
  В `sales` записуються лише одні загальні дати `warranty_start` та `warranty_end`. Але при покупці кількох позицій (наприклад, iPhone + чохол) на девайс має бути 12 місяців гарантії, а на чохол — 1 місяць. В базі немає поштучного відстеження гарантійних залишків для кожного елемента чеку (`sale_items`).

---

### 2️⃣ Constraint Guardian (Безпека, продуктивність та ліміти)
* **N+1 запити в мутаціях:**  
  Для кожного товару в кошику виконується 2–3 SQL-запити (зчитування stock + запис stock + списання). При кошику з 5 товарів це генерує 15 послідовних запитів до API Supabase. Це створює навантаження на мережу та сильно уповільнює Server Action.
* **Транзакційність на рівні СКБД:**  
  Supabase JS SDK не дозволяє відкривати транзакції (`BEGIN ... COMMIT`) безпосередньо. Єдиний надійний архітектурний спосіб вирішити проблему N+1 та гарантувати 100% атомарність чеку — це перенесення логіки списання та створення продажу в єдину PostgreSQL функцію (RPC) на боці бази даних.

---

### 3️⃣ User Advocate (UX, зручність та когнітивне навантаження)
* **Контроль ціни нижче собівартості:**  
  Хоча система має валідацію, продавець може випадково або навмисно встановити роздрібну ціну (`unit_price`) нижче собівартості (`unit_cost`), принісши збиток бізнесу. Потрібно додати візуальне попередження в POS-кошику, якщо ціна продажу падає нижче собівартості.
* **Візуальні плейсхолдери:**  
  У разі відсутності фото товару (`photo_urls` є порожнім), в Bento-блоках каталогу замість порожнього місця мають рендеритися красиві плейсхолдери з гармонійними кольорами (HSL) та іконками відповідно до типу товару (телефон, чохол, кабель тощо).
* **Покращення навігації "Назад":**  
  При виборі категорії Bento-блоки товарів відкриваються в тій самій області. Кнопка "Назад" має ставати візуально більшою та помітнішою для швидкого виходу в загальний список категорій, відповідно до вимог користувача.

## 🚦 3. Висновки Арбітра (Integrator Verdict)

**Статус аудиту:** **APPROVED** (із доопрацюванням)

**Обґрунтування:**  
Логіка Bento POS є стабільною, проте вимагає усунення потенційного дублювання продажу техніки (IMEI) та покращення візуальної частини для відповідності преміум-стандартам (Bento UI, плейсхолдери, збільшена кнопка "Назад" та попередження про собівартість).

### Кроки для втілення в поточному спринті:
1. Додати перевірку статусу `in_stock` (Optimistic locking) для девайсів у `createMultiSaleAction`.
2. Додати візуальне попередження про ціну нижче собівартості в POS кошику.
3. Реалізувати збільшену акцентну кнопку "Назад" у Bento POS каталозі.
4. Додати красиві HSL-плейсхолдери для товарів без фото.
